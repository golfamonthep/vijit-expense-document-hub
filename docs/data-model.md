# Data Model

## แนวคิดหลัก

data model ของระบบนี้เป็น `document-first domain model`

ระบบจะไม่เริ่มจาก object ประเภท expense โดยตรง แต่จะเริ่มจาก `Document` ที่รับเข้ามาจากหลายช่องทาง แล้วค่อยเชื่อมโยง, extract, match, review, approve และนำไปสร้าง report

เอกสารฉบับนี้เป็น conceptual data model เท่านั้น ยังไม่ใช่ SQL schema

## Design Principles

- `Document` คือ first-class entity
- `ExpenseCase` ใช้เป็นตัวรวมหลักฐานหลายรายการ
- AI output ถูกเก็บแยกจาก human-approved business truth
- approval action ต้องแยกเป็นเหตุการณ์ที่ audit ได้
- รองรับ single-company usage ในระยะแรก แต่ไม่ผูก model ให้ใช้ได้แค่บริษัทเดียว

## Core Entities

### `Company`

Purpose:

- แทนบริษัทเจ้าของข้อมูลและ workflow

Conceptual fields:

- company identity
- legal name
- display name
- reporting preferences
- active status

Relationships:

- หนึ่ง `Company` มีหลาย `Document`
- หนึ่ง `Company` มีหลาย `ExpenseCase`
- หนึ่ง `Company` มีหลาย `AccountingReport`

หมายเหตุ:

- ระยะแรกอาจใช้งานเพียงบริษัทเดียว แต่ model ควรมี `Company` ตั้งแต่ต้น

### `LineUser`

Purpose:

- เก็บตัวแทนของผู้ส่งข้อมูลจาก LINE หรือ channel user profile ที่เกี่ยวข้อง

Conceptual fields:

- company reference
- channel user identifier
- display name
- source channel type
- status

Relationships:

- หนึ่ง `LineUser` อาจเชื่อมกับหลาย `Document`

หมายเหตุ:

- แม้ชื่อ entity จะเป็น `LineUser` ตาม requirement ปัจจุบัน แต่ model ควรรองรับการเพิ่ม channel user แบบอื่นในอนาคตได้

### `Document`

Purpose:

- แทนไฟล์หรือข้อความที่เข้าระบบในฐานะหลักฐานต้นทาง

Conceptual fields:

- company reference
- source channel
- source message or event reference
- uploader or sender reference
- file reference or text payload reference
- mime type or document type hint
- capture timestamp
- processing status
- review status

Relationships:

- หนึ่ง `Document` อาจมีหลาย `DocumentExtraction`
- หนึ่ง `Document` ควรมีความเป็นสมาชิกอยู่ในศูนย์หรือหนึ่ง canonical `ExpenseCase` ในช่วงเวลาหนึ่ง ผ่าน `ExpenseCaseDocument`
- หนึ่ง `Document` สร้าง `AuditLog` ได้หลายเหตุการณ์

หมายเหตุ:

- แม้จะใช้ `ExpenseCaseDocument` เป็น relationship entity เพื่อเก็บ metadata ของการจับคู่ แต่ในเชิงธุรกิจ `Document` ไม่ควรถูกนำไปใช้เป็นหลักฐานของหลาย canonical `ExpenseCase` พร้อมกัน เว้นแต่มีการ re-match หรือแก้ไขโดยมนุษย์

### `DocumentExtraction`

Purpose:

- เก็บผล AI extraction และ raw output ของแต่ละ `Document`

Conceptual fields:

- document reference
- extractor version or provider reference
- extracted fields
- raw extraction output
- confidence summary
- warnings
- created timestamp

Relationships:

- หลาย `DocumentExtraction` อาจอ้างถึง `Document` เดียวกัน

หมายเหตุ:

- แยก entity นี้ออกจาก `Document` เพื่อให้เก็บหลายรอบ extraction และเก็บ raw output ได้

### `ExpenseCase`

Purpose:

- แทนชุดค่าใช้จ่ายหนึ่งกรณีที่อาจประกอบด้วยหลาย `Document`

Conceptual fields:

- company reference
- case status
- business date
- description
- normalized vendor
- normalized amount
- category
- reviewer summary
- approval readiness indicators

Relationships:

- หนึ่ง `ExpenseCase` เชื่อมกับหลาย `Document` ผ่าน `ExpenseCaseDocument`
- หนึ่ง `ExpenseCase` มีหลาย `ApprovalAction`
- หนึ่ง `ExpenseCase` อาจถูกอ้างถึงในหลาย `AccountingReport`

### `ExpenseCaseDocument`

Purpose:

- เป็น relationship entity ระหว่าง `ExpenseCase` และ `Document`

Conceptual fields:

- expense case reference
- document reference
- document role in case
- sort or display order
- matching confidence or matching reason

Relationships:

- ใช้รองรับ many-to-many ที่มี metadata กำกับ

หมายเหตุ:

- ใน implementation ภายหลัง ควรตีความความยืดหยุ่นของ relationship นี้เพื่อรองรับ audit, re-match และ metadata ของการจับคู่ มากกว่าการอนุญาตให้ `Document` เดียวเป็นหลักฐานของหลาย canonical `ExpenseCase` พร้อมกันตาม business rule ปกติ

### `ApprovalAction`

Purpose:

- บันทึกเหตุการณ์ review, approve, reject หรือ decision อื่นที่มีผลต่อ `ExpenseCase`

Conceptual fields:

- expense case reference
- actor reference
- action type
- action timestamp
- comment or reason
- state before
- state after

Relationships:

- หลาย `ApprovalAction` อ้างถึง `ExpenseCase` เดียวกันได้
- แต่ละ action ควรเชื่อมโยงกับ `AuditLog` ได้

### `AccountingReport`

Purpose:

- แทนรายงานทางบัญชีรายรอบ เช่น รายงานรายเดือน

Conceptual fields:

- company reference
- reporting period
- report type
- report status
- generated timestamp
- generated by

Relationships:

- หนึ่ง `AccountingReport` อ้างถึงหลาย approved `ExpenseCase`
- หนึ่ง `AccountingReport` มีหลาย `ReportExport`

หมายเหตุ:

- report เป็น downstream artifact ไม่ใช่ source of truth แทน `ExpenseCase` หรือ `Document`
- ความสัมพันธ์ระหว่าง `AccountingReport` กับ approved `ExpenseCase` อาจถูกแทนด้วย relationship entity เพิ่มเติมในระยะ schema design หากต้องเก็บ membership หรือ snapshot metadata ของรายงาน

### `ReportExport`

Purpose:

- เก็บผลการ export รายงาน เช่น Word หรือ PDF ในอนาคต

Conceptual fields:

- accounting report reference
- export format
- storage reference
- export status
- created timestamp

Relationships:

- หลาย `ReportExport` อาจอ้างถึง `AccountingReport` เดียวกัน

### `AuditLog`

Purpose:

- บันทึกเหตุการณ์สำคัญเพื่อใช้ตรวจสอบย้อนหลัง

Conceptual fields:

- company reference
- actor reference
- entity type
- entity reference
- event type
- event payload summary
- created timestamp

Relationships:

- `AuditLog` อาจอ้างถึง `Document`, `ExpenseCase`, `ApprovalAction`, `AccountingReport` หรือ entity อื่นตาม event

### `IntegrationConnection` (optional)

Purpose:

- เก็บการตั้งค่าระดับ conceptual สำหรับ external integrations

Conceptual fields:

- company reference
- integration type
- connection status
- configuration reference
- last sync metadata

Relationships:

- หนึ่ง `Company` อาจมีหลาย `IntegrationConnection`

หมายเหตุ:

- entity นี้ยังเป็น optional เพราะการตั้งค่า integration จริงยังไม่อยู่ใน scope ปัจจุบัน

## Conceptual Relationships Summary

- `Company` 1:N `Document`
- `Company` 1:N `ExpenseCase`
- `Company` 1:N `AccountingReport`
- `Document` 1:N `DocumentExtraction`
- `ExpenseCase` M:N `Document` ผ่าน `ExpenseCaseDocument`
- `ExpenseCase` 1:N `ApprovalAction`
- `AccountingReport` 1:N `ReportExport`
- `AuditLog` เชื่อมแบบ event reference ไปยังหลาย entity

## Lifecycle States

### `Document` Lifecycle

ตัวอย่างสถานะระดับ conceptual:

- received
- stored
- queued for extraction
- extracted
- needs review
- matched
- archived

### `ExpenseCase` Lifecycle

ตัวอย่างสถานะระดับ conceptual:

- draft
- needs review
- ready to approve
- approved
- rejected
- exported

### `AccountingReport` Lifecycle

ตัวอย่างสถานะระดับ conceptual:

- draft
- generated
- exported
- superseded

## Single-Company Now, Multi-Company Later

แนวทางรองรับทั้งสองช่วง:

- ระยะแรกสามารถ configure ให้ใช้งานสำหรับบริษัทเดียวได้
- แต่ทุก core entity ควรมี conceptual relation กับ `Company`
- permission, storage partitioning และ reporting scope ควรถูกออกแบบให้เพิ่มหลายบริษัทได้ภายหลังโดยไม่ต้องเปลี่ยนแนวคิดหลักของ model

## SQL Schema Baseline Notes

The repository now includes an initial Supabase migration at `supabase/migrations/20260612230000_create_document_first_schema.sql` and seed data at `supabase/seed.sql`.

This first SQL schema keeps the conceptual model document-first and translates it into these concrete decisions:

- `documents` is the required intake root. Every uploaded file becomes a document record before extraction or case review.
- `document_extractions` stores AI output separately from human-reviewed accounting truth.
- `expense_cases` stores the reviewed accounting case and uses a simple `month` text field in `YYYY-MM` format for monthly grouping.
- `expense_case_documents` is the many-to-many link that allows multiple evidence documents for one expense case.
- `accounting_reports` is downstream-only and is intended to be generated from approved `expense_cases`.

Additional SQL-level choices in this baseline:

- Check constraints are used for workflow statuses, types, and roles instead of PostgreSQL enums so future changes can be migrated more easily.
- Indexes are added for `company_id`, `month`, `status`, `document_type`, `source_type`, `line_user_id`, `expense_case_id`, `document_id`, and `transfer_ref` where those fields exist.
- `updated_at` trigger support is added for tables that include an `updated_at` column.
- Row Level Security is enabled on all public tables as a safe Supabase default. Policies are intentionally deferred until the auth and permission model is implemented.
